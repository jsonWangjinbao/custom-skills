# design-to-code v2 优化设计

> **日期**: 2026-07-02
> **状态**: 设计已确认
> **版本**: v2.0 (Draft)

## 1. 背景与目标

### 1.1 现状问题

当前 `design-to-code` skill (v1) 是一个聚焦 HTML→RN/Web 代码精准还原的单向转换器，存在以下局限：

| 问题 | 表现 |
|------|------|
| **线性流程** | 只有从 HTML 到代码的单向管道，不支持需求分析→设计→执行的完整链路 |
| **单输入** | 只能处理一份 HTML + 可选截图，不支持多页面批量输入 |
| **无状态管理** | 不支持断点续作，中断后需从头开始 |
| **无功能点跟踪** | 没有功能点列表，无法验证需求是否全部实现 |
| **无用户确认环节** | 直接出代码，缺少功能点确认、设计审核等中间确认节点 |
| **无增量/重构区分** | 无法处理需求分析和原代码调研的场景 |

### 1.2 优化目标

将 design-to-code 从"HTML→代码转换器"升级为**全链路需求交付管道**，支撑两种工作模式：

- **增量需求**：在现有 Web/RN/H5 上加功能，需求描述 + HTML+截图 → 实现
- **重构需求**：H5→RN、RN→RN、Web→Web 重构，原代码分析还原功能 → 再还原样式

### 1.3 核心原则

- **功能点列表是一切行动的真理源**
- **UI 是锦上添花，不阻塞功能开发**
- **所有状态由文档持久化，天然支持中断恢复**
- **每一步完成双向更新：执行步骤 ✅ → 对应功能点 ✅**
- **后续修改从功能点列表开始，同步更新技术文档**

## 2. 整体架构

### 2.1 文档驱动管道

```
  [1. 输入收集] → [2. 功能点分析] → [3. UI审计] → [4. 技术设计] → [5. 执行 & 勾选] → [6. 自测验证]
                                                                                             
        所有状态持久化到: 输入目录/.design-to-code/ 或 ai-wiki/【需求名】/.design-to-code/
```

### 2.2 管道阶段总览

| 1 | 功能点分析 | 需求描述/PRD + 原代码(重构)/现有代码(增量) | `features.md` | ✅ 必须确认 |

| 4 | 执行 & 勾选 | tech-design.md | `execution.md` + 代码 | — |
| 5 | 自测验证 | features.md | 验证报告 | ✅ 最终确认 |

## 3. 详细步骤设计

### 3.1 STEP 0: 启动 & 初始化

询问需求名称 → 自动创建需求文件夹。

**流程：**
1. Skill: "请为这次需求输入一个名称："
2. 用户输入名称（如 "FSMS-证件管理 APP交互体验优化"）
3. Skill 检查 `ai-wiki/【名称】/` 是否存在
4. 不存在 → 创建 `ai-wiki/【名称】/` 及其 `.design-to-code/` 子目录
5. 存在 → 检测是否有未完成的执行文档，提示是否恢复进度

**关于 ai-wiki 路径：**
- 优先使用 `/Users/wangjinbao/wanchen-code/ai-wiki/` 作为基础路径
- 该目录不存在时，回退到当前项目根目录

### 3.2 STEP 1: 功能点分析

核心步骤。根据需求模式分流。

#### 1.1 模式选择

```
Skill: "这次是增量需求还是重构需求？"

增量 → 在当前代码仓库分析相关模块 + 需求描述 → 生成功能点
重构 → 先分析原始代码功能点 + 需求描述 → 生成功能点
```

#### 1.2a 增量模式

1. 用户提供需求描述或产品文档路径
2. Skill 读取需求内容并理解范围
3. Skill 提问涉及哪些现有模块，读取相关代码理解现有结构
4. 结合需求 + 现有代码 → 生成 `features.md`
5. 展示给用户确认

#### 1.2b 重构模式

1. 用户提供需求描述或产品文档路径
2. Skill 提问原始代码路径
3. Skill 深入分析原始代码，**尤其关注各组件间的联动关系**：

   - **页面/路由结构**：页面切换、参数传递、路由守卫
   - **组件树**：父子组件层级、兄弟组件通信
   - **组件联动**（重点）：
     - 组件 A 的值变化 → 组件 B 的状态/数据变化
     - 跨表单字段联动：字段 A 选中值 → 字段 B 的选项列表更新
     - 校验规则联动：字段 A 的值范围 → 字段 B 的校验规则动态变化
     - 组件显隐联动：组件 A 的状态 → 组件 B 显示/隐藏/禁用
     - 表单之间的联动：表单 A 提交成功后 → 表单 B 刷新或预填充
   - **数据流和状态管理**：Context/Redux/Zustand 的状态变更链路
   - **API 调用**：请求参数、响应处理、错误处理、接口依赖链
   - **样式方案**：主题变量、布局方案、响应式规则

   > ⚠️ 联动关系分析是重构的关键所在。每识别一个联动关系，在功能点列表中用一个独立的子项记录，以便后续实现时不会遗漏。

4. 结合需求 + 原代码分析 → 生成 `features.md`
   - 功能点标注来源：`来自原代码` / `需求新增`
5. 展示给用户确认

#### 1.3 `features.md` 格式

```markdown
# 功能点列表 — 【需求名称】

> 生成时间: 2026-07-02
> 需求模式: 增量 / 重构

## 功能点清单

| ID | 模块 | 功能描述 | 来源 | UI完整 | 联动/依赖 | 状态 | 备注 |
|----|------|---------|------|--------|----------|------|------|
| F-001 | 证件上传 | 调用相机/相册选择证件照片 | 原代码 | ⬜ | F-002 上传后触发 | 待开发 | |
| F-002 | 证件上传 | 上传证件照片到服务器 | 原代码 | ⬜ | F-001 选择后启用 | 待开发 | |
| F-003 | 证件列表 | 展示已上传证件列表 | 需求新增 | ⬜ | F-002 上传成功后刷新列表 | 待开发 | PRD §2.1 |

> 联动关系说明：每项依赖指**当被依赖功能点发生变化/完成时，当前功能点需要响应**。例如 F-003 依赖 F-002，表示 F-002 上传成功后 F-003 列表需刷新。

## 功能点与 UI 对照

| 功能点 | 关联 HTML | 关联截图 | UI 覆盖情况 |
|--------|----------|---------|------------|
| F-001 ~ F-002 | 待补充 | 待补充 | ⬜ |
| F-003 | 待补充 | 待补充 | ⬜ |
```

### 3.3 STEP 2: UI 材料收集 & 审计（可选）

用户确认功能点后进入。UI 材料对功能开发**非必需**，用户可以跳过。

#### 2.1 询问

```
Skill: "请提供 HTML 文件 + 对应截图的目录路径，我会自动扫描并对照功能点生成 UI 完整性报告。"
```

#### 2.2 扫描

Skill 自动扫描目录，按文件名模式匹配 HTML 和截图：

```
page-a.html  +  page-a.png     → 配对成功 ✅
page-b.html  +  (无截图)        → 配对不完整 ⚠️
(无 HTML)    +  feature-x.png  → 孤图，询问意图
```

#### 2.3 生成 `ui-audit.md`

```markdown
# UI 完整性审计报告

## 功能点对照

| ID | 功能 | HTML | 截图 | 状态 |
|----|------|------|------|------|
| F-001 | 证件上传 | upload.html ✅ | upload.png ✅ | 完整 |
| F-002 | 上传服务 | upload.html ✅ | upload.png ✅ | 完整 |
| F-003 | 证件列表 | list.html ✅ | ⬜ 缺失 | 不完整 |
| F-004 | 证件详情 | ⬜ 缺失 | ⬜ 缺失 | 缺失 |
```

#### 2.4 跳过 UI

Skill 询问用户：

```
"部分功能点缺少 UI 材料。是否跳过 UI 还原阶段，直接进入技术设计？
- 是 → 标记 'UI 待补充'，功能按项目通用样式实现
- 否 → 补充文件后再继续
- 以后再补 → 标记 'UI 待补充'，后续可重新启动 skill 进入 UI 完善模式
```

### 3.4 STEP 3: 技术设计文档

#### 3.1 输入

- `features.md` ✅（必须）
- `ui-audit.md` ⚪（可选）
- 项目现有代码风格和架构（如果有）

#### 3.2 输出 `tech-design.md`

```markdown
# 技术设计 — 【需求名称】

## 1. 概述
- 本次改动范围
- 涉及模块
- 目标平台

## 2. 组件架构
├── components/
│   ├── DocumentUploader      // F-001
│   ├── OCRPreview            // F-002
│   └── ...

## 3. 数据流
- 状态管理方案
- API 接口定义
- 缓存策略

## 4. 路由设计
- 新增路由
- 路由参数

## 5. UI 说明
- 有 UI 材料 → 注明参考的 HTML+截图
- 无 UI 材料 → 使用项目通用样式

## 6. 功能点对照
| ID | 组件 | 文件路径 |
|----|------|---------|
| F-001 | DocumentUploader | pages/upload/ |

## 7. 关注点
- 组件库 gotchas（引用 references/gotchas/）
- 跨平台差异
```

#### 3.3 用户审核

用户审阅后提出修改意见或批准通过。

### 3.5 STEP 4: 执行 & 勾选

#### 4.1 生成 `execution.md`

```markdown
# 执行文档 — 【需求名称】

## 执行步骤

### 第一阶段: 基础组件
- [ ] Step 1: 创建 DocumentUploader 组件  → F-001 | 30min
- [ ] Step 2: 实现 OCR 预览              → F-002 | 20min

### 第二阶段: 列表与详情
- [ ] Step 3: 创建列表组件                → F-003 | 25min

### 第三阶段: 样式打磨（UI 待补充）
- [ ] Step 4: UI 还原                   → F-001~F-003

### 第四阶段: 自测
- [ ] Step 5: 自测                      → 全部功能点
```

#### 4.2 分步执行

Skill 按顺序逐一执行步骤，每个步骤：

1. 读取技术文档中对应部分
2. 读取项目代码风格
3. 生成/修改代码
4. 写入文件
5. 更新 `execution.md`：步骤标记 ✅
6. 更新 `features.md`：关联功能点标记 ✅
7. 展示变更摘要

#### 4.3 中断恢复

下次启动时 Skill 检测 `.design-to-code/execution.md`：

```
检测到未完成的执行文档：
  需求: 【需求名】
  已完成: Step 1~3 (F-001 ~ F-003 ✅)
  待完成: Step 4~9 (F-004 ~ F-007 ⬜)

是否继续执行？[Y/n]
如果有新的变更需求请先说明变更内容。
```

### 3.6 STEP 5: 自测验证

所有步骤完成后自动进入验证：

```
逐项检查 features.md 功能点：
  F-001 证件上传 ✅  组件存在，API 调用正常
  F-002 上传服务 ✅  API 方法已实现
  ...

全部功能点已实现。自测通过。
UI 样式还原尚未完成（因缺少 HTML+截图）。
建议补充 UI 材料后再次启动 skill 进入样式完善模式。
```

### 3.7 后续修改

```
用户提出变更 → Skill 检查 features.md
  → 询问是否更新功能点列表
  → 确认后在 features.md 上新增/修改功能点（标记"需求变更"）
  → 同步更新 tech-design.md
  → 更新 execution.md 插入新步骤
  → 按新执行文档继续
```

## 6. 关键决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 状态持久化方式 | 文档驱动（.md 文件） | 人类可读，天然支持中断恢复，无需数据库 |
| 功能点 ID 方案 | F-001 递增 | 简洁，与执行步骤双向关联 |
| 文档存储位置 | 输入目录/.design-to-code/ | 与需求内容同目录，避免跨项目引用 |
| UI 可跳过 | 是 | 不阻塞功能开发核心路径 |
| 增量/重构分流时机 | STEP 1 开始 | 早分流避免后续路径混淆 |
| 提交确认机制 | 仍要求用户确认 | 遵守 CLAUDE.md 第 13 条 |
