# 黑盒封装组件 — 识别与补偿方法论（平台通用）

> 本文定义平台无关的识别方法与处理流程。各平台的具体组件名单、不可控样式表与代码示例见平台目录下的 gotchas（如 RN: `reference/rn/gotchas/component-library/blackbox-wrapper-component.md`）。

## 什么是黑盒封装组件

组件库中「封装了表单容器 + 原生控件」的高阶组件（例如 RN 的 `CommonFormItem`、H5/PC 的封装 Form.Item 类组件）。其内部组合了 label、控件、错误提示等复合结构，行高、间距、对齐方式、分隔线等布局由组件库默认样式决定，使用者无法通过 props 精确控制。

## 识别条件

同时满足以下条件即视为黑盒组件：

1. 一个组件同时渲染 label、输入控件、校验提示等复合结构
2. 内部布局维度（行高 / 分隔线 / 对齐 / 控件尺寸）不暴露 props 或暴露不完整
3. 设计稿对这些维度有精确数值或视觉要求

## 常见不可控样式维度

| 样式维度              | 黑盒组件默认行为   | 常见设计稿要求       |
| --------------------- | ------------------ | -------------------- |
| 行高                  | 由容器组件默认决定 | 固定行高（如 48px）  |
| 字段间分隔线          | 通常无             | 1px 分隔线           |
| label-value 布局      | 组件库固定对齐方式 | 可能要求不同对齐     |
| 控件尺寸              | 固定宽高           | 设计稿可能有不同尺寸 |
| 辅助元素（箭头/图标） | 可能不渲染         | 设计稿通常有明确指示 |

## 处理流程（四阶段挂接）

| 阶段   | 动作                                                                                                                        |
| ------ | --------------------------------------------------------------------------------------------------------------------------- |
| audit  | 在 ui-audit.md 的「组件库渲染差异分析」章节中，对每个使用黑盒组件的 UI 元素标注差异项和补偿方案                             |
| design | 在 tech-design.md 的「组件选择决策表」中，明确每个元素使用黑盒组件还是自定义实现                                            |
| build  | 选择黑盒组件必须在代码中落实补偿方案（如手动插入分隔线、覆写样式）；设计稿要求精确控制时优先用原生容器 + 原子控件自定义实现 |
| verify | 将渲染结果与设计截图做 side-by-side 对比，确认黑盒组件的默认样式不产生偏差                                                  |

## 决策原则

- 设计稿无精确行高 / 分隔线 / 对齐要求 → 使用黑盒组件（开发效率高，行为一致）
- 设计稿有精确行高、字段间分隔线、特殊对齐 → 优先原生容器 + 原子控件自定义实现
- 选择完全自定义时，需自行处理表单值转换逻辑（取值 / 回显 / 校验触发），参考黑盒组件源码中对应类型的实现
