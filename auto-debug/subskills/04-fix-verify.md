# Phase 4 - 修复、验证循环与最终清理

## 进入条件

- 用户已确认 Phase 3 的修复方案
- 或 Phase 0 高匹配知识复用经用户确认（此路径无注入物，跳过服务器相关步骤，验证与清理只针对业务改动）

## 任务：应用修复

1. **执行修复**：应用确认的代码修改
2. **保留注入日志**（dbg 调用不删除）— 用于用户重新验证
3. **重启日志服务器**：
   - kill 旧进程，清空旧日志
   - 重新启动 server.mjs
   - 更新 manifest.json 中 serverPid
4. 告知用户：「✅ 修复已应用，请重新操作验证，完成后告诉我『操作完了』或『修好了』」

## 任务：验证循环

用户反馈后：

- 「操作完了」/「还是不行」→ 回到 [03-collect.md](03-collect.md) 重新收集分析（服务器保持运行，dbg 注入不删除）
- 「修好了」/「可以了」/「没问题」→ 执行下方最终清理

此循环可能进行多轮：Phase 3 → Phase 4 → Phase 3 → ... 直到用户确认修复完成。

## 任务：最终清理

仅在用户确认修复完成后执行（「修好了」「没问题了」「清理吧」等）：

1. **杀掉日志服务器**：`kill <PID>`
2. **清理注入代码**：
   - 根据 manifest.json 的 `before` 字段精确还原每个注入点（用 Edit 工具反向替换）
   - 删除 `src/utils/claudeDebug.ts`
   - 删除整个 `.claude/auto-debug/` 目录（manifest 随目录一并删除）
3. **验证清理**：`grep -r "dbg"` 确认无调试代码残留；确认临时文件已删除
4. 告知用户：「✅ 调试代码已清理，仅保留业务修复改动」

## 出口

- 清理完成 → 必须进入 [05-distill.md](05-distill.md)（不可跳过）
